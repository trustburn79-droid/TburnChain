3. Trust Score-Based BFT Consensus

TBURN Chain introduces an innovative consensus mechanism that extends traditional Byzantine Fault Tolerance with a dynamic trust scoring system. This approach reduces the impact of malicious or underperforming validators while rewarding consistent, high-quality participation.

3.1 Trust Score Calculation

Each validator maintains a Trust Score (TS) calculated from multiple performance metrics:

Trust Score Formula
TS = α·Uptime + β·BlockProduction + γ·VoteAccuracy + δ·StakeRatio + ε·NetworkContribution

Where the coefficients are:

  • α = 0.25: Validator uptime percentage (99%+ optimal)
  • β = 0.20: Block production success rate
  • γ = 0.20: Consensus vote accuracy and timeliness
  • δ = 0.20: Stake amount relative to network total
  • ε = 0.15: Cross-shard message relay and network health contribution

3.3 Block Production

Validators are selected for block production through a weighted random selection algorithm where the probability is proportional to their Trust Score. This ensures high-performing validators produce more blocks while maintaining decentralization. 

// Block Proposer Selection Algorithm
function selectBlockProposer(validators, blockHeight) {
  const weightedPool = validators.map(v => ({
    address: v.address,
    weight: v.trustScore * v.stakeAmount
  }));
  
  const seed = hash(blockHeight + previousBlockHash);
  const totalWeight = sum(weightedPool.map(v => v.weight));
  const selection = seed % totalWeight;
  
  return findValidatorByWeight(weightedPool, selection);
}

3.4 Finality Guarantee

TBURN Chain achieves single-slot finality through a two-phase commit process:

  1. Propose Phase (50ms): Selected proposer broadcasts block to shard validators
  2. Commit Phase (50ms): Validators vote, 2/3+ weighted consensus confirms finality

Total finality time: 100ms per block, with cross-shard finality achieved within 1 second.

4.4 Cross-Shard Communication

Cross-shard transactions utilize an asynchronous message passing protocol with the following characteristics:

  • Message Queue: Priority-based routing with O(1) shard pair selection
  • Batch Processing: Messages aggregated into batches for efficiency
  • Proof Generation: Merkle proofs enable trustless cross-shard verification
  • Caching: 30-second TTL shard state cache for optimal performance

4.5 Automatic Scaling

The sharding engine monitors network load in real-time and triggers scaling events when thresholds are exceeded:

  • Scale Up: When average shard load exceeds 70% for 5 minutes
  • Scale Down: When average shard load drops below 30% for 15 minutes
  • Rebalance: When load imbalance exceeds 30% between shards
